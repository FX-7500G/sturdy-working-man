# GitLab CI/CD: Решение проблем с Docker Registry и деплоем в Kubernetes

## Проблема (Issue)
При развёртывании пайплайна в GitLab CI возникали критические ошибки:  
1. Stage Build: Docker не мог залогиниться в реестре. Ошибка `http: server gave HTTP response to HTTPS client` из-за чего сборка образа падала 
2. Stage Deploy: Ошибка авторизации кластера `Kube-context not found` и kubectl из CI не подключался к кластеру
3. Legacy код: Использовались устаревшие пути, захардкожены IP-адреса и логины, неверные менеджеры пакетов

## Что изначально было:
- Использовался устаревший CI конфиг (legacy), содержащий хардкод и несовместимые зависимости

## Анализ
- Реестр был настроен на работу по HTTP, хотя Docker использует HTTPS по умолчанию
- Образы раннеров были на Alpine, хотя менеджеры были `apt`
- Некорректный вариант интеграции K8s через серты
- K8s не был корректно проброшен в переменные CI
- DinD стартовал не сразу из-за чего следующие команды выполнялись раньше времени

--- 

## Полное решение
#### 1. Обновление безопасности
- Вычистил хардкод из конфига гита (`.gitlab-ci.yml`). Все чувствительные данные перенёс в переменные. Это упростило правки кода и увеличило безопасность секретов

#### 2. Фикс Docker registry
- Добавил кастомный CA сертификат:
```bash
apk add --no-cache openssl
echo | openssl s_client -showcerts -connect <IP>:5005 2>/dev/null | openssl x509 -outform PEM > /etc/docker/certs.d/<IP>:5005/ca.crt
```

#### 3. Ожидание готовности Docker
```bash
until docker info; do sleep 2; done
```
- Использовал её, потому что пайплайн не дожидался инициализации докера и в итоге выполнял последующие команды которые приводили к ошибке `Cannot connect to the Docker daemon at unix:///var/run/docker.sock. Is the docker daemon running?`

#### 4. Оптимизация Docker build
```bash
DOCKER_DRIVER=overlay2
DOCKER_BUILDKIT=1
```
- Вручную указал современный драйвер для производительности, потому что есть риски, что DinD откатится на vfs  
- По такой же причине указал и BUILDKIT, чтобы докер точно работал на современных движках  

#### 5. Настройка k8s доступа
- Добавил `KUBECONFIG` в переменные GitLab
- Проверил доступ через `kubectl config view`

#### 6. Отладка логов
- Для диагностики проблемы подключения, используем `kubectl get nodes -v=10`. Это помогло мне найти ошибку в конфигурации прокси 

#### 7. Добавлен entrypoint: [""]
- Без него команды не выполнялись корректно

## Результат
- CI/CD pipeline теперь стабильно собирает docker-образы
- Образы публикуются в локальный реестр
- Деплой в K8s выполняется автоматически через GitLab CI
- Устранены ошибки с TLS, доступом к registry и подключения к кластеру
- Стабилизировал выполнение pipeline

## Выводы
Узнал довольно много чего...
- Docker registry по HTTP требует явной настройки доверия (CA-сертификатов)
- Kubernetes доступ удобнее передавать через KUBECONFIG
- entrypoint может влиять на выполнение CI job
- Нужно задать готовность демона, чтобы пайплан не ломался на первых секундах (`until docker info`)
- Всегда стоит переводить на перменные и шаблоны, а то хардкод это почти никогда не хорошо..)
- CI/CD может быть как и средой отладки. Использовал его для анализа логов и решения серьёзных ошибок, что помогло глубже понять разницу между окружениями, зависимостями и правами доступа
